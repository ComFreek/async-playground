expectTimelyIn

  • expectTimelyIn(promise: PromiseLike<any>, millis: number, upperBuffer?: number, lowerBuffer?: number): Promise<void>
    • Defined in timing.common-spec.ts:126

    Expect a promise to be resolved in about millis milliseconds.

    Tiny time buffers are required to accommodate various effects (inexact timeouts in general, GC, the cost of this very function call etc.).

    The promise is expected using expectTimelyWithin to resolve in the time frame [max(millis - lowerBuffer, 0), millis + upperBuffer].

    Parameters

    • promise: PromiseLike<any>
    • millis: number
    • Default value upperBuffer: number = 5
    • Default value lowerBuffer: number = 5

    Returns Promise<void>

    A void promise which is

           - resolved iff. the given promise is resolved in the time frame
         explained above. In this case, the fulfillment occurs right
         after the given promise is resolved.
       - rejected iff. the given promise is not resolved in the time
         frame. In this case, the rejection occurs after
         `(millis + upperBuffer)` ms.

expectTimelyWithin

  • expectTimelyWithin(promise: PromiseLike<any>, minMillis: number, maxMillis: number): Promise<void>
    • Defined in timing.common-spec.ts:52

    Expect a promise to be fulfilled within a time frame starting withe the call to this function.

    Beware that due to JS' execution model, it cannot be guaranteed that timeouts can be exactly met. Users are therefore encouraged to provide some time buffer (e.g. 10 - 15 ms) to accomodate this effect.

    see

    expectTimelyIn

    Parameters

    • promise: PromiseLike<any>

      The PromiseLike object. Being only a {@link PromiseLike}, rejections cannot be handled as the interface is lacking a .catch method.

    • minMillis: number

      The minimum number of milliseconds (an integer >= 0) the promise is expected to remain unresolved. Specifying 0 means that no lower limit will be imposed.

    • maxMillis: number

      The maximum number of milliseconds (an integer >= 0), the promise is expected to remain unresolved. Users can make reasonably sure (depending on the use case) that a promise never resolves if they specify a value high enough.

    Returns Promise<void>

    A void promise, which is

          - fulfilled iff. the given promise resolved in the time frame.
        In this case, the fulfillment occurs right after the given
        promise is resolved.
      - rejected iff. the given promise did not resolve in the time
        frame. In this case, the rejection occurs either right after
        the given promise is resolved before the `minMillis` timeout
        or right after the `maxMillis` timeout.
